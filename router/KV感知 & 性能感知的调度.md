## 3、Dynamo的调度算法

Cost Function = overlap_score_weight * prefill_blocks + decode_blocks

其中，
- prefill_blocks：需要执行prefill计算的blocks数量（排除已命中缓存的blocks）
- decode_blocks：根据请求的输入tokens长度和每个推理实例的输出序列估算得出
- overlap_score_weight：可调权重，平衡缓存命中优化和负载分布，较高的权重有利于缓存重用（优化TTFT），而较低的权重则优先考虑均匀的负载分配（提高ITL）

## 4、Preble实现

Preble的E2调度算法在收到新请求$R_k$时，调度器首先在全局前缀树中做前缀匹配，找到最长匹配的前缀节点。然后：

**判断条件：**

> 若「匹配到的前缀 token 数」>「剩余未匹配 token 数」→ 选择 **Exploitation（利用）**
> 否则 → 选择 **Exploration（探索）**

**解释：** 如果能节省的重计算（匹配部分）比需要新增的计算（未匹配部分）更多，那么"复用缓存"的收益大于"负载均衡"的收益，就应该优先复用。反之则应该先考虑把请求分散到更轻松的GPU。

当选择 Exploration 时，E2需要为每块 GPU $i$ 计算一个综合代价：$L_i + M_i + P_i$，选代价最低的GPU：

Cost Function = $L_i + M_i + P_i$

这三部分的含义如下：

**$L_i$（历史负载代价）**

在过去的时间窗口 $H$（默认3分钟）内，GPU $i$ 处理的所有请求的计算时间之和：

$$L_i = \sum_{r \in W} (PT_r + DT_r)$$

其中 $PT_r$ 是请求 $r$ 的 prefill 时间估算（按照未命中缓存的token数回归估算），$DT_r$ 是decoding时间估算（按历史平均输出长度估算）。

> 用历史窗口而非实时负载，是因为KV缓存的放置效果是长期的——今天把某个前缀缓存在GPU1上，未来几分钟内到来的相关请求都会受益。

**$M_i$（驱逐代价）**

GPU通常处于满载状态，为了给新请求腾出显存，需要驱逐一些KV缓存。被驱逐的缓存越热（被更多请求命中），驱逐代价越高：

$$M_i = \sum_{j \in E} PT_j \times N_j$$

其中 $E$ 是需要驱逐的节点集合，$PT_j$ 是节点 $j$ 对应的计算时间，$N_j$ 是该节点的命中率（历史窗口内命中该节点的请求比例）。

> **实例：** GPU2上缓存了一个被 60% 请求命中的热点节点，GPU3 上缓存了一个只被 5% 请求命中的冷节点。相同计算量下，驱逐 GPU2 上的节点代价是 GPU3 的 12 倍，E2 会倾向于把新请求发给 GPU3。

**$P_i$（新请求本身的计算代价）**

在 GPU $i$ 上运行新请求 $R_k$ 时，需要重新计算的 token 数对应的 prefill 时间（已命中缓存的 token 不计代价）

## 5、DualMap实现

DualMap的核心目标是：**同时做到"缓存亲和性（prefix affinity）"和"性能均衡（load balancing）"**，而不是像 Preble/Dynamo那样在两者之间做权衡取舍。

它的解法不是一个单一cost function，而是一个**三层递进的设计**：先用哈希保证流量收敛，再用双候选做负载均衡，最后用热点感知处理极端情况。

**背景：缓存亲和性与负载均衡的矛盾**

- **缓存亲和性**要求：相同前缀的请求，必须路由到同一个实例，这样那个实例上的KV缓存才能被复用；
- **负载均衡**要求：请求要分散到多个实例，不能让某台机器过载；

这两个需求直接冲突——如果完全按前缀路由，热点前缀会把单台机器打爆；如果完全按负载路由，缓存命中率近乎为零。

传统做法只能在二者之间做权衡（如Dynamo的overlap_score_weight），但总是牺牲其中一方。

DualMap的思路是：不做权衡，而是换一个灵活的解法。

**核心思路：把"单点固定"变成"小组固定"**

DualMap 的关键洞察是：

> 缓存亲和性不要求"必须发到同一个实例"，只要求"必须发到有缓存的实例"。

如果我们能保证某个前缀的所有流量，**始终只在固定的两个实例之间分配**，那么随着时间推移，这两个实例上都会建立起该前缀的KV缓存，此后不管请求发到哪一个，都能命中缓存。

这样，负载均衡的空间从全部N个实例缩小到固定的2个候选，缓存亲和性的要求也从固定1个实例放宽到固定2个实例中选一个，两个目标同时成立。

**Step 1：双哈希候选对（Dual Hashing → Candidate Pair）**

对每个请求，调度器根据其前缀构建一个前缀键$K$，然后用两个不同的哈希函数映射到两个实例：

$$i_1 = f_1(K), \quad i_2 = f_2(K)$$

得到候选对 $(i_1, i_2)$：
- **相同前缀 → 永远映射到同一个候选对**（哈希函数确定性）
- **不同前缀 → 大概率映射到不同候选对**（哈希分散性）

然后使用 **Power-of-Two-Choices（PoTC）** 策略，在候选对中选负载更低的实例。

**Step 2：动态前缀键长度（前缀热度树）**

上面的双哈希方案还有一个残余问题：**候选对只有2个实例**，当某个前缀流量极大时，2个实例也可能同时过载，PoTC彻底失效。

DualMap引入「前缀热度树」来解决这个问题。核心思路是：**动态调整哈希键$K$的粒度**，让热点前缀自然映射到更多的候选对上：
- **前缀冷门时**：使用较短的 key（粗粒度），相似请求共享同一个候选对 → **流量聚合，提升缓存复用率**
- **前缀热门时**：延长 key（细粒度），微小差异的请求映射到不同候选对 → **流量分散，避免过载**

调度器维护一棵前缀树，每个节点记录流量占比 $\rho$。当 $\rho$ 超过阈值，自动延伸哈希键深度，把一个热点前缀的流量拆分给多个候选对共同承担。

> 本质是：**通过控制缓存亲和性的强度，间接控制负载分散的程度**。热点时亲和性松一点，允许流量分散；冷门时亲和性紧一点，保证缓存聚合。

**Step 3：SLO-aware实例选择**

```
优先级1：能否满足SLO（TTFT / ITL不超限）
优先级2：当前负载（queue depth / token）
```

具体逻辑：
- 两个候选都满足SLO → 选负载更低的
- 只有一个满足SLO → 直接选它，不看负载
- 两个都不满足SLO → 选违约程度最小的

## 6、LMetric

LMetric的调度器在收到新请求时，为每个推理实例 $i$ 计算一个综合分数，选分数最低的实例：

$$ Cost Function = \text{Score}(i) = \text{P-token}(i) \times \text{BS}(i)$$

其中：
- $\text{P-token}(i)$：请求发到实例 $i$ 时，需要重新计算的 prefill token 数（总输入长度 − KV 缓存命中的 token 数）
- $\text{BS}(i)$：实例 $i$ 当前需要处理的请求总数（running batch size + waitting batch size）

**两个指标的选取逻辑**

**P-token（而非KV命中率）**

KV命中率是相对值，对长请求和短请求一视同仁，P-token 是绝对值，直接反映需要执行的计算量。

> **实例：** 请求 A 长度 1000 token，命中率 90%，需要计算 100 token；请求 B 长度 100 token，命中率 80%，需要计算 20 token。命中率看 A 更优，但实际上 B 的计算代价更小（100 > 20）。P-token 能正确区分这一差异。

**BS（而非Total Tokens）**

总token数（输入+输出的累计长度）是历史账，已经入队的token无法撤回，BS是当前时刻的并发压力，每新增一个请求就多一个decode步骤在争抢算力，对TPOT的影响是即时的。
